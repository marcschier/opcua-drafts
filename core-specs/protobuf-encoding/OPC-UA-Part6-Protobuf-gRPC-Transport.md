# OPC UA Part 6 — Protobuf gRPC TransportProtocol

**Working draft for submission to the OPC Foundation Working Group**  
**Proposed addition to:** OPC 10000-6 Mappings v1.05.07 clause 7 TransportProtocols  
**Namespace:** `http://opcfoundation.org/UA/` (base OPC UA namespace)  
**Version:** 0.1.0 · **Date:** 2026-07-03

> **Status — working draft.** This document defines rules for carrying OPC UA Service calls over gRPC with Protobuf bodies. It does not define a complete Service catalog; the single `Read` example illustrates the mapping rule that all Services follow.

---

## 1 Scope

This TransportProtocol maps OPC UA Services defined in OPC 10000-4 to gRPC methods. Request and response bodies are proto3 messages that encode the corresponding OPC UA Service request and response Structures using the OPC UA Protobuf DataEncoding defined in clause 5.6. The transport is intended for client/server Service calls, including subscription-style Services, and is not a PubSub message mapping.

## 7.6 OPC UA gRPC

### 7.6.1 General

An OPC UA gRPC Endpoint shall expose OPC UA Services as gRPC methods. Each OPC UA Service request Structure and response Structure shall be represented as a proto3 message generated by the OPC UA Protobuf DataEncoding rules in clause 5.6. The generated request and response messages are the gRPC method input and output types.

The URL scheme for this TransportProtocol shall be:

```text
opc.grpc://<host>[:<port>][/<path>]
```

The TransportProfileUri for this TransportProtocol shall be:

```text
http://opcfoundation.org/UA-Profile/Transport/grpc-protobuf
```

The gRPC content type shall be `application/grpc+proto`. Servers may reject other gRPC content types for this TransportProfileUri.

### 7.6.2 Service-to-method mapping

Each OPC UA Service defined in OPC 10000-4 shall map to one gRPC method with the same service semantics:

```proto
rpc <ServiceName>(<ServiceName>Request) returns (<ServiceName>Response);
```

Services whose OPC UA semantics produce a continuing sequence of notifications, such as Publish/subscription-style interactions, may use gRPC streaming. A profile shall specify whether such a Service is client-streaming, server-streaming or bidirectional streaming. Unary request/response semantics shall remain the default mapping for Services that complete with one response.

The request and response message names, packages and field mappings shall be stable parts of the service contract. A receiver shall not infer a different field order or wrapper shape from the gRPC method name; it shall use the DataEncoding rules and the shared contract descriptors.

### 7.6.3 SecureChannel, Session and request metadata

The gRPC connection provides the byte transport. OPC UA SecureChannel and Session semantics remain visible at the Service layer:

- The Session authentication token from `RequestHeader.authenticationToken` shall be sent as binary gRPC metadata key `opcua-session-authentication-token-bin` using the OPC UA Protobuf `NodeId` representation.
- The complete `RequestHeader` shall be sent as binary gRPC metadata key `opcua-request-header-bin` using the OPC UA Protobuf representation of `RequestHeader`. If the request body also contains `request_header`, the metadata and body values shall match.
- `RequestHeader.requestHandle` shall also be sent as ASCII metadata key `opcua-request-handle` to permit routing, tracing and response correlation before body decoding.
- SecureChannel correlation data, when a deployment profiles a SecureChannel token over gRPC, shall be sent as ASCII metadata keys `opcua-securechannel-id` and `opcua-securechannel-token-id`.
- Sequence numbering for ordered request streams shall use ASCII metadata key `opcua-sequence-number`. Unary calls may omit it when HTTP/2 stream ordering and request handles are sufficient.

Servers shall reject a call with `UNAUTHENTICATED` if the token is missing or invalid, `PERMISSION_DENIED` if the Session lacks permission, and `INVALID_ARGUMENT` if duplicated metadata and body header fields disagree.

### 7.6.4 Service errors and ServiceFault

A successful OPC UA Service response with a Good service result shall be returned as a normal gRPC response message.

If OPC UA Service processing returns a `ServiceFault` or another service-level Bad StatusCode, the server shall return a non-OK gRPC status. The gRPC status code shall reflect the transport-visible class of error, the status message should contain the symbolic OPC UA StatusCode name when available, and the status details shall carry the raw OPC UA StatusCode value. The detail message shall include at least:

```proto
message OpcUaStatusDetail {
  fixed32 status_code = 1;
  optional string symbolic_id = 2;
  optional opcua.protobuf.v1.DiagnosticInfo diagnostic_info = 3;
}
```

The detail may be carried with the standard gRPC rich error mechanism (`grpc-status-details-bin`) or an equivalent profile-defined binary metadata field. Application-level per-operation results inside a successful response, such as `ReadResponse.results[*].status_code`, remain in the response body and shall not be converted to gRPC status.

### 7.6.5 Schema and contract identity

The service `.proto` file and its transitive `FileDescriptorSet` are the shared contract for a gRPC Endpoint. The SchemaId shall be computed with the same canonical transitive-closure SHA-256 algorithm defined for the OPC UA Protobuf DataEncoding.

Clients shall know the service contract before decoding a response. A server shall advertise the SchemaId in initial metadata key `opcua-schema-id`. Clients may send the expected SchemaId in request metadata. If both are present and differ, the receiver shall fail the call with `FAILED_PRECONDITION`.

The service contract shall be registered in the xRegistry catalog. The catalog entry shall bind the SchemaId to the service package, method set, canonical `FileDescriptorSet`, TransportProfileUri and compatible OPC UA namespace/model versions. A client that does not have the descriptor locally shall retrieve it from the catalog by SchemaId and verify the hash before decoding messages.

### 7.6.6 Worked example: Read

The following reduced service contract is illustrative. The field names and nested messages model the OPC UA `ReadRequest` and `ReadResponse` Structures; omitted fields follow the same DataEncoding rules.

```proto
syntax = "proto3";

package opcua.service.v1;

import "opcua_builtins.proto";

service OpcUaServices {
  rpc Read(ReadRequest) returns (ReadResponse);
}

message RequestHeader {
  opcua.protobuf.v1.NodeId authentication_token = 1;
  sfixed64 timestamp = 2;
  uint32 request_handle = 3;
  uint32 return_diagnostics = 4;
  optional string audit_entry_id = 5;
  uint32 timeout_hint = 6;
}

message ResponseHeader {
  sfixed64 timestamp = 1;
  uint32 request_handle = 2;
  fixed32 service_result = 3;
  opcua.protobuf.v1.DiagnosticInfo service_diagnostics = 4;
}

message ReadValueId {
  opcua.protobuf.v1.NodeId node_id = 1;
  uint32 attribute_id = 2;
  optional string index_range = 3;
}

message ReadRequest {
  RequestHeader request_header = 1;
  double max_age = 2;
  uint32 timestamps_to_return = 3;
  repeated ReadValueId nodes_to_read = 4;
}

message DataValueResult {
  opcua.protobuf.v1.DataValue value = 1;
}

message ReadResponse {
  ResponseHeader response_header = 1;
  repeated DataValueResult results = 2;
  repeated opcua.protobuf.v1.DiagnosticInfo diagnostic_infos = 3;
}
```

A client that calls `OpcUaServices/Read` shall:

1. Encode `ReadRequest` with clause 5.6 Protobuf rules.
2. Send `opcua-session-authentication-token-bin`, `opcua-request-header-bin`, `opcua-request-handle` and, when known, `opcua-schema-id` metadata.
3. Use `application/grpc+proto`.
4. Decode a normal `ReadResponse` body using the same service contract.
5. Treat a non-OK gRPC status with `OpcUaStatusDetail` as the mapped OPC UA `ServiceFault` or service-level error.

All other OPC UA Services follow the same rule: generated request Structure message, generated response Structure message, one gRPC method, and StatusCode-carrying gRPC status for service-level faults.

## 8 Insertion into OPC 10000-6 v1.05.07

Insert this TransportProtocol as new clause **7.6 OPC UA gRPC** after **7.5 WebSockets**. The child clauses shall be:

- **7.6.1 General**
- **7.6.2 Service-to-method mapping**
- **7.6.3 SecureChannel, Session and request metadata**
- **7.6.4 Service errors and ServiceFault**
- **7.6.5 Schema and contract identity**
- **7.6.6 Worked example: Read**

Add the TransportProfileUri `http://opcfoundation.org/UA-Profile/Transport/grpc-protobuf`, the URL scheme `opc.grpc`, and content type `application/grpc+proto` to the TransportProfileUri and endpoint URL scheme guidance for OPC UA gRPC.
